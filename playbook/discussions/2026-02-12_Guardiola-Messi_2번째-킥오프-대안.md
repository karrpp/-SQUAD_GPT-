## 2. Guardiola -- 시장 분석 & 후보 3개 도출

### 시장 컨텍스트

**2026년 마이크로 SaaS 환경:**
- SaaS 산업 $300B 돌파. 마이크로 SaaS는 $15.7B -> $59.6B(2030) 성장 전망.

**성공 패턴 분석:**
- AirTrackBot: 1인 개발, ~$7K MRR
- StageTimer.io: 이벤트 전문, ~$8.3K MRR
- HelpKit: 1년 미만에 $5K MRR 돌파
- 공통점: **한 가지 불편을 확실히 해결**

#### 후보 A: "AI Agent Playbook" -- AI 에이전트 팀 빌더

**한 줄 요약:** 성민 FC처럼 AI 에이전트 팀을 구성하고 운영할 수 있는 플랫폼

**아이디어 상세:**
- 사용자가 프로젝트에 맞는 AI 에이전트 "스쿼드"를 구성
- 각 에이전트에 역할/성격/프롬프트를 부여
- 워크플로우(킥오프 -> 실행 -> 검증)를 시각적으로 설계
- 실행 시 각 에이전트가 순서대로 답변 생성
- 핵심: "프롬프트 1개"가 아니라 "팀 단위 프롬프트 오케스트레이션"

**Guardiola 분석:**

| 항목 | 분석 |
|------|------|
| **PMF 가능성** | 높음. Claude Code, GPT Codex 사용자 급증. "프롬프트 하나"로는 한계를 느끼는 사용자 증가 |
| **기술 난이도** | 중간. Claude/OpenAI API 호출 체이닝 + 프롬프트 관리 UI. 성민이 이미 SQUAD_GPT로 경험 보유 |
| **수익화** | Freemium. 무료: 3 에이전트. Pro($15/mo): 무제한 에이전트 + 고급 워크플로우. Team($49/mo): 공유 |
| **경쟁** | n8n, Dify 등 워크플로우 도구 존재. 하지만 "에이전트 팀 + 성격 부여" 컨셉은 차별화 |
| **6개월 MVP** | 가능. Phase 1(2개월): 에이전트 생성 + 순차 실행. Phase 2(2개월): 워크플로우 시각화. Phase 3(2개월): 마켓플레이스 |
| **월 1,000명** | 도전적. AI 개발자/크리에이터 타깃. ProductHunt + AI 커뮤니티(Reddit, X) |

**리스크:**
- Claude/OpenAI API 의존도 높음 (정책 변경 리스크)
- "프롬프트 오케스트레이션"을 원하는 사용자가 충분한가?
- Anthropic, OpenAI가 직접 이 기능을 만들 수 있음

#### 후보 B: "Claude Code Snippets" -- AI 코딩 세션 기록/공유 도구

**한 줄 요약:** AI와 코딩하면서 나온 좋은 패턴, 프롬프트, 코드 스니펫을 저장/검색/공유하는 도구

**아이디어 상세:**
- 개발자가 Claude Code / Cursor / Copilot과 작업 중 "이거 좋다!" 싶은 순간을 캡처
- 프롬프트 + AI 응답 + 최종 코드를 한 세트로 저장
- 태그, 언어, 프레임워크별 분류
- 커뮤니티 공유: 다른 개발자의 "AI 코딩 레시피" 검색
- 핵심: "개인 AI 코딩 노트" + "커뮤니티 레시피북"

**Guardiola 분석:**

| 항목 | 분석 |
|------|------|
| **PMF 가능성** | 중상. 92% 개발자가 AI 도구 사용, 51%가 매일 사용. "좋은 프롬프트/패턴 저장하고 싶다"는 니즈 존재 |
| **기술 난이도** | 낮음-중간. CRUD + 검색 + 소셜. Next.js + Supabase로 빠른 개발 가능 |
| **수익화** | Freemium. 무료: 50개 스니펫. Pro($9/mo): 무제한 + 비공개. Team($29/mo): 팀 공유 |
| **경쟁** | GitHub Gist, Snipd 등 존재. 하지만 "AI 코딩 세션" 특화는 없음 |
| **6개월 MVP** | 충분. Phase 1(6주): CRUD + 태깅. Phase 2(6주): 검색 + 커뮤니티. Phase 3(6주): 브라우저 확장 |
| **월 1,000명** | 현실적. 개발자 커뮤니티(Reddit r/programming, HackerNews, X)에서 바이럴 가능 |

**리스크:**
- "GitHub Gist로 충분하지 않나?" 반론
- AI 코딩 트렌드가 바뀌면 수요 감소 가능
- 개발자는 무료 도구에 익숙 (과금 저항)

#### 후보 C: "Ship Log" -- 1인 개발자/크리에이터를 위한 빌드인퍼블릭 대시보드

**한 줄 요약:** 혼자 만드는 사람이 개발 과정을 기록하고, 진행 상황을 공개하며, 초기 사용자를 모으는 도구

**아이디어 상세:**
- 1인 개발자가 매일 "오늘 한 것" 기록 (마크다운)
- 자동으로 타임라인/로드맵 시각화
- 공개 페이지: 잠재 사용자가 진행 과정을 지켜봄
- "관심 등록" 버튼 -> 런칭 대기 리스트 수집
- GitHub 커밋 연동 -> 자동 진행 상황 업데이트
- 핵심: **"빌드인퍼블릭"을 구조화된 도구로 만든 것**

**Guardiola 분석:**

| 항목 | 분석 |
|------|------|
| **PMF 가능성** | 중간. 인디해커 커뮤니티에서 #buildinpublic 트렌드 강함. 하지만 X(Twitter)만으로 충분하다는 의견도 존재 |
| **기술 난이도** | 낮음. 마크다운 에디터 + 타임라인 시각화 + GitHub API. 성민 스택으로 2-3개월 내 MVP |
| **수익화** | Freemium. 무료: 1 프로젝트. Pro($12/mo): 무제한 프로젝트 + 커스텀 도메인 + 분석 |
| **경쟁** | Indie Page, Logsnag 등 부분적 경쟁. 하지만 "빌드인퍼블릭 올인원"은 없음 |
| **6개월 MVP** | 여유. 기술 난이도가 낮아 3-4개월이면 런칭 가능 |
| **월 1,000명** | 도전적. 인디해커는 작은 커뮤니티. 확장하려면 크리에이터/유튜버까지 타깃 확대 필요 |

**리스크:**
- TAM(전체 시장)이 작음. 인디해커/솔로 파운더는 한정된 시장
- "X 스레드로 충분"한 사용자가 많음
- 월 $12 과금 의향이 낮을 수 있음

## 3. 양자 토의 -- 핵심 쟁점

### 쟁점 1: "성민이 직접 쓰는가?" 테스트

**Messi:**
> 세 후보에 "성민이 매일 쓸 것인가?" 질문을 던져보자.
>
> - **후보 A (Agent Playbook)**: 쓴다. 성민이 지금 SQUAD_GPT에서 매일 하는 것의 확장.
> - **후보 B (Code Snippets)**: 쓴다. Claude Code로 작업하면서 "이 패턴 저장하고 싶다" 매일 느낌.
> - **후보 C (Ship Log)**: 쓴다. 빌드인퍼블릭을 하고 싶지만 X만으로는 부족함을 느낌.
>
> 세 개 다 통과. 하지만 강도가 다르다.
> **A >= B > C** 순으로 "진짜 필요함"이 강하다.

**Guardiola:**
> 동의한다. 하지만 "성민이 쓰는 것"과 "남들도 쓰는 것"은 다르다.
> 여기서 PMF 신호를 봐야 한다.
>
> - **후보 A**: "AI 에이전트 팀"을 원하는 사람이 얼마나 되는가? 얼리어답터 한정일 수 있다.
> - **후보 B**: 92%의 개발자가 AI 코딩 도구를 쓴다. 시장이 넓다. 그리고 "좋은 프롬프트/패턴 저장"은 보편적 니즈.
> - **후보 C**: 인디해커 커뮤니티는 열정적이지만 작다. 확장성 우려.

**합의:**
- 시장 크기: **B > A > C**
- 개인 필요도: **A >= B > C**

### 쟁점 2: "6개월 내 월 1,000명" 현실성

**Guardiola:**
> 현실적으로 분석하겠다.
>
> **후보 A (Agent Playbook):**
> - 타깃: AI 파워유저, 개발자, 크리에이터
> - 도달 채널: ProductHunt, AI 서브레딧, X
> - 월 1,000명 난이도: **상** (니치가 좁음. 워크플로우 도구 경쟁 치열)
>
> **후보 B (Code Snippets):**
> - 타깃: AI 코딩 도구 사용 개발자 (전체 개발자의 92%)
> - 도달 채널: HackerNews, Reddit r/programming, DEV.to, ProductHunt
> - 월 1,000명 난이도: **중** (개발자 커뮤니티 바이럴 가능. "이 프롬프트 패턴 봐라!" 공유 본능)
>
> **후보 C (Ship Log):**
> - 타깃: 인디해커, 1인 개발자
> - 도달 채널: Indie Hackers, X #buildinpublic
> - 월 1,000명 난이도: **상** (작은 시장. 확장 어려움)

**Messi:**
> Guardiola 분석이 정확하다. 후보 B가 가장 넓은 시장에 닿을 수 있다.
> 하지만 한 가지 더. **후보 A는 성민만의 이야기를 담을 수 있다.**
>
> "나는 AI 에이전트 5명으로 팀을 만들어서 프로덕트를 만들고 있습니다.
> 그 시스템을 누구나 쓸 수 있게 만들었습니다."
>
> 이건 **스토리**가 된다. 스토리는 바이럴의 연료다.

**Guardiola:**
> 맞다. 하지만 스토리만으로 1,000명은 안 된다. 제품이 좋아야 한다.
> 후보 A의 문제는 **"에이전트 오케스트레이션"의 복잡도**다.
> 일반 사용자가 에이전트 프롬프트를 직접 짜야 한다면 진입장벽이 높다.

**Messi:**
> 그래서 "템플릿"이 필요하다. 성민 FC가 직접 쓰는 SQUAD 시스템을 템플릿으로 제공하면,
> 사용자는 복사해서 바로 쓴다. 마치 Notion 템플릿처럼.

**합의:**
- 월 1,000명 현실성: **B > A > C**
- 단, 후보 A는 스토리 마케팅 강점 보유

### 쟁점 3: "성민의 차별화가 있는가?"

**Messi:**
> 여기가 핵심이야, 집중하자.
>
> - **후보 A**: 차별화 **강함**. 성민은 이미 SQUAD_GPT를 운영하고 있다. 이 경험 자체가 해자(moat).
> - **후보 B**: 차별화 **보통**. 누구나 만들 수 있다. CRUD + 검색. 기술적 해자 없음.
> - **후보 C**: 차별화 **보통**. 컨셉은 좋지만 기술적 진입장벽 낮음.
>
> 후보 B는 "좋은 제품"이 될 수 있지만, **"성민만이 만들 수 있는 제품"**은 아니다.
> 후보 A는 성민만이 만들 수 있다. 왜? **실제로 AI 에이전트 팀을 돌려본 경험**이 있으니까.

**Guardiola:**
> Messi의 지적이 날카롭다. 차별화 관점에서는 후보 A가 압도적이다.
>
> 하지만 전략적 리스크도 지적한다:
> - Anthropic이 Claude에 "에이전트 팀" 기능을 넣으면? (실제로 Claude Code가 이미 그 방향)
> - OpenAI가 "GPT 팀" 기능을 출시하면?
>
> **플랫폼 리스크**가 크다.

**Messi:**
> 그건 모든 AI 도구가 가진 리스크다. 하지만 우리가 가진 건 **"실전 경험에서 나온 워크플로우 설계"**다.
> Anthropic이 기능을 만들어도, "어떤 에이전트를 어떤 순서로 배치하면 좋은 결과가 나오는가?"는
> 실전 경험이 없으면 모른다. 그게 우리 해자다.

**합의:**
- 차별화: **A >> B >= C**
- 플랫폼 리스크: A가 가장 높지만, 실전 노하우로 방어 가능

---

### 쟁점 4: "수익화 구조"

**Guardiola:**
> 수익화 현실성을 비교한다.
>
> **후보 A (Agent Playbook):**
> - $15/mo Pro, $49/mo Team
> - 1,000명 x $15 = $15,000 MRR (최대)
> - AI API 비용 전가 필요 (사용자가 자기 API 키 사용)
> - 수익 구조: 건강함. 하지만 API 키 설정이 온보딩 장벽
>
> **후보 B (Code Snippets):**
> - $9/mo Pro, $29/mo Team
> - 1,000명 x $9 = $9,000 MRR (최소)
> - AI API 비용 거의 없음 (저장/검색만)
> - 수익 구조: 매우 건강. 마진 90%+
>
> **후보 C (Ship Log):**
> - $12/mo Pro
> - 1,000명 x $12 = $12,000 MRR
> - 인프라 비용 최소
> - 수익 구조: 건강하지만 과금 의향이 낮을 수 있음

**Messi:**
> 수익만 보면 B가 가장 안전하다. 하지만 **"성장 잠재력"**은 A가 더 크다.
> A는 $49 Team 플랜이 가능하다. 기업/팀 시장까지 확장하면 ARPU가 훨씬 높다.

**합의:**
- 안전한 수익: **B > C > A**
- 성장 잠재력: **A > B > C**

## 6. 최종 추천안

### 추천: 후보 A -- "AI Agent Playbook"

**한 줄:** AI 에이전트를 팀처럼 구성하고 운영하는 플랫폼. 성민 FC의 SQUAD 시스템을 제품화.

**왜 이것인가:**

1. **성민만의 해자**: 실제로 AI 에이전트 팀을 운영한 경험. 이 경험에서 나온 워크플로우 설계가 핵심 자산.
2. **Dog-fooding 가능**: 성민이 매일 SQUAD_GPT로 쓰고 있는 것. 본인이 첫 번째 사용자.
3. **스토리 마케팅**: "AI 5명으로 팀을 만들어 프로덕트를 개발했습니다" -- 이 스토리 자체가 마케팅.
4. **성장 잠재력**: 개인 $15/mo -> 팀 $49/mo -> 기업 커스텀. ARPU 확장 가능.
5. **피벗 가능**: 실패 시 후보 B(Code Snippets)로 기술 스택 유지한 채 전환.

### Phase 0 계획 (2주)

**목표:** "AI 에이전트 팀" 컨셉이 사람들에게 통하는가?

**산출물:**
- 성민 FC SQUAD 시스템 데모 영상 (3분)
- "SQUAD_GPT로 만든 실제 결과물" 비교 (프롬프트 1개 vs 에이전트 5명)
- 5명에게 공유 후 반응 수집

**성공 기준:**
- 5명 중 3명 이상: "이 컨셉을 이해한다" + "쓰고 싶다"
- 5명 중 2명 이상: "돈 낼 의향 있다"
- 5명 중 1명 이상: "이거 공유하고 싶다"

**실패 시:**
- 후보 B(Code Snippets)로 피벗
- Phase 0에서 만든 기술 자산 재활용

### 기술 스택 (변경 없음)

- Next.js 14+ (App Router) + TypeScript
- TailwindCSS + shadcn/ui
- Supabase (인증 + DB)
- Vercel (배포)
- Claude API / OpenAI API (에이전트 실행)

### 리스크 & 대응

| 리스크 | 발생 확률 | 대응 |
|--------|-----------|------|
| 컨셉 이해 실패 (너무 어려움) | 중간 | 30초 온보딩 영상 + "성민 FC" 기본 템플릿 |
| 플랫폼 리스크 (Anthropic/OpenAI가 같은 기능 출시) | 중간 | 워크플로우 설계 노하우 = 방어 해자. 속도로 승부 |
| 시장이 작음 (얼리어답터 한정) | 중간 | 후보 B로 피벗. 개발자 시장은 넓음 |
