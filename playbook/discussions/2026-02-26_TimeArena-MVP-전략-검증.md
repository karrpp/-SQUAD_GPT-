## Messi의 방향 검증

### 방향

Time Arena의 핵심 아이디어는 좋지만, 모바일 네이티브 앱이 아닌 PWA(Progressive Web App)로 MVP를 시작해야 한다. 우리 팀의 무기로 싸워야 이긴다.

### 판단 근거

- "사용자에게 진짜 도움이 되는가?" → **YES**. 시간 인식을 바꿔주는 컨셉 자체는 진짜 도움이 된다.
- 우리 팀의 무기는 **React + Vite**다. 이미 React 19, Vite 7이 세팅되어 있다. React Native/Flutter로 가면 학습 비용이 3개월 일정을 잡아먹는다.
- 1인 + AI 팀의 강점은 **속도와 반복**이다. PWA는 웹 배포로 즉시 사용자에게 닿을 수 있다.

### 팀 실행 가능성 판단

**네이티브 모바일 앱 3개월 MVP: 위험하다.**
모바일 네이티브 개발 경험이 검증되지 않은 상태에서 React Native/Flutter + 인앱 결제 + 푸시 알림 + 앱스토어 심사까지 3개월에 넣는 건 과부하다.

**PWA 기반 MVP 3개월: 충분히 가능하다.**
React로 타이머 엔진, Canvas/CSS 기반 마을 시각화, localStorage 또는 간단한 백엔드로 데이터 저장. 이건 기존 역량 안에 있다. 단, 월2 친구 기능은 백엔드(Firebase/Supabase)가 필요 — @Modric에게 기술 판단 위임 필요.

### 핵심 리스크 (Messi)

| 리스크 | 심각도 | 설명 |
|--------|--------|------|
| 수동 타이머 입력 피로도 | **치명적** | 매번 입력하는 행위 자체가 마찰. Day7 리텐션 20% 목표는 이 마찰 극복 여부에 달림 |
| 마을 시각화 완성도 | **높음** | 시각적 보상 퀄리티가 낮으면 일반 타이머 앱과 차별화 불가 |

### Guardiola에게 전달 (현장 인사이트)

1. **플랫폼 전략을 PWA-first로 재설정**하라. 네이티브는 PMF 검증 이후.
2. **월2 친구 기능 제거**. 대신 마을 시각화 완성도와 알림 시스템에 집중.
3. **기술 스택 의사결정을 @Modric에게 위임** (PWA 백엔드, 마을 렌더링, 오프라인 지원).
4. **MVP 성공 기준 축소**: "타이머를 3일 연속 사용한 사용자 비율"이 핵심 지표.

## Guardiola의 전략 프레임

### 전략 프레임 — 3가지 갭 발견

**갭 1 — 역량 불일치**: 팀은 React 웹 역량만 검증됨. 모바일 네이티브 앱은 빌드 환경, 스토어 심사, 푸시 알림 등 완전히 다른 복잡도.

**갭 2 — 범위 과잉**: 소셜 기능(백엔드 인프라)과 인앱 결제(Apple/Google 정책 대응)를 1인 팀이 3개월에 소화 불가.

**갭 3 — 핵심 가치 검증 부재**: MVP에서 먼저 검증해야 할 것은 "수동 타이머로 시간 절약 행위가 실제로 반복되는가?"다. 마을 테마, 스킨 스토어, 친구 기능은 그 이후 문제.

### 기술 스택 판단

**권고: PWA (React 19 + Vite 기반)**

| 옵션 | 판단 | 이유 |
|------|------|------|
| **PWA** | ✅ 권고 | 기존 역량 활용, 스토어 심사 없음, 즉시 배포 |
| React Native | ❌ 제외 | 숨겨진 복잡도 (Xcode, Android Studio, 디바이스 디버깅) |
| Flutter | ❌ 제외 | Dart 새로 학습 필요. MVP 단계 이점 없음 |

**전환 시점**: PWA로 핵심 루프 검증 + DAU 1,000 이상 달성 후 네이티브 전환 고려.

### Scope 재조정

| 구분 | 항목 |
|------|------|
| ✅ 유지 | 타이머 엔진, 마을 시각화(서울 1개), 로컬 스토리지, PWA 설치, 온보딩, 일간/주간 대시보드 |
| ❌ 제거 | 친구 기능, 인앱 결제, 스킨 스토어, 부산/제주/파리/뉴욕 테마, A/B 테스트 |
| ⬇️ 축소 | 푸시 알림(P2 격하), 랜드마크 언락 → 건물 레벨업으로 단순화 |
| ➕ 추가 고려 | 로컬 스토리지 기반 시작, 30초 이내 온보딩, 마을 스크린샷 SNS 공유 |

### 우선순위

**P0 (필수 — 이것 없으면 앱이 아님)**
- 타이머 핵심 엔진 (설정, 실행, 조기 종료, 포인트 변환)
- 마을 성장 시각화 (서울 테마, 기본 건물 5종)
- 로컬 스토리지 데이터 영속화
- PWA manifest + Service Worker
- 온보딩 플로우

**P1 (중요 — 리텐션 직결)**
- 일간/주간 통계 대시보드
- 건물 레벨업 시스템
- 마을 스크린샷 공유

**P2 (선택 — 검증 후 추가)**
- 푸시 알림 (PWA 기반)
- 다크모드
- 추가 테마
- 소셜/친구 기능
- 수익화

### 리스크 & 대응 (Guardiola)

| 리스크 | 보고서 포함 | 대응 |
|--------|------------|------|
| 수동 타이머 피로도 → 2주 내 이탈 | ✅ | 원탭 프리셋(15분/30분/1시간), 최근 기록 자동 제안 |
| 마을 성장 보상감 약함 | ✅ | "마을 확인 빈도"를 핵심 지표로 추적. 반복 없으면 재설계 |
| PWA iOS 제약 (푸시, 설치 유도) | ❌ 누락 | iOS 16.4+ 지원, 초기 유저는 Android 중심으로 |
| **1인 팀 번아웃** | ❌ 누락 | 6주 MVP(P0) → 2주 테스트 → 4주 개선으로 버퍼 확보 |
| 경쟁앱 포지셔닝 (Forest, Opal) | ❌ 누락 | "집중"(Forest) vs "여가 조절"(Time Arena) 차별화. 온보딩에서 명확히 전달 |
| 마을 비주얼 에셋 제작 역량 부재 | ❌ 누락 | 픽셀 아트 스타일 + AI 이미지 생성 도구 활용. 초기엔 CSS 미니멀로 시작 |
